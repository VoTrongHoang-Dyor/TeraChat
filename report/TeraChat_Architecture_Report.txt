PHẦN 1: MESH NETWORK & TỐI ƯU iOS
1.1  Phạm vi và Use Case thực tế
Mesh Network giải quyết một window rất hẹp nhưng có giá trị cao trong môi trường doanh nghiệp: mất internet trong khi LAN hoặc không gian vật lý vẫn hoạt động. Khi Mac Mini mất điện, các thiết bị client — iPhone, laptop, Android — vẫn còn pin và có thể hình thành mạng P2P trực tiếp mà không cần anchor node.

Điểm mạnh so với BitChat trong môi trường doanh nghiệp: TeraChat đã có sẵn identity infrastructure (Ed25519 key per device trong Secure Enclave) và MLS session state cached trên từng thiết bị. CRDT DAG trong hot_dag.db cho phép merge delta khi Mac Mini restore — lượng message cần sync lại rất nhỏ do cúp điện thường chỉ vài phút đến vài chục phút.

1.3  Unified Transport Abstraction — Rust Core
Để tránh duplicate logic giữa hai transport path, cần một unified abstraction tại tầng Core:

trait MeshTransport: Send + Sync {
    async fn send(&self, peer: &PeerId, payload: &[u8]) -> Result<()>;
    async fn recv(&self) -> Result<(PeerId, Vec<u8>)>;
    fn estimated_throughput_kbps(&self) -> u32;
}

// WifiDirectTransport impl MeshTransport  (Tier 1, < 60m)
// BleRelayTransport impl MeshTransport    (Tier 2, > 60m)

MLS E2EE và CRDT sync layer gọi vào MeshTransport mà không cần biết transport phía dưới — đây là nguyên tắc Deep Module giúp tránh coupling giữa crypto layer và network transport.

1.5  Workflow kích hoạt Mesh
Cơ chế kích hoạt cần cân bằng giữa UX và tránh false positive:

	•	Tự động: Grace period 30 giây sau khi mất internet trước khi chuyển sang Mesh mode — tránh flicker khi mạng thoáng mất
	•	Thủ công: Nút "Bật Mesh" trong Settings — dành cho scenario cố ý (họp ngoài trời, vùng tín hiệu yếu)
	•	Pop-up xin phép: Chỉ lần đầu tiên per device. Sau đó lưu preference. Cần giải thích rõ: Mesh dùng BLE và Wi-Fi Direct, có thể ảnh hưởng pin.
	•	UI indicator: Hiển thị rõ ràng khi đang ở Mesh mode (Dark Navy background theo Design System) — user cần biết latency và throughput khác bình thường
	•	Sync khi restore: Khi Mac Mini restore, CRDT merge delta tự động. Không cần user action. Thời gian sync tỷ lệ với số message trong window offline — thường < 5 giây


PHẦN 2: BẢO MẬT — MitM & MÃ HÓA KHÁNG LƯỢNG TỬ
2.1  Phân tích rủi ro MitM trong Mesh Mode
Threat model trong Mesh thay đổi đáng kể so với internet. Attacker không còn ngồi trên đường truyền internet mà là thiết bị giả mạo trong cùng không gian vật lý. Điều này thực ra dễ phòng thủ hơn với điều kiện MLS E2EE đã được implement đúng.

Phân tích theo lớp bảo mật hiện tại:

	•	Content: An toàn tuyệt đối — MLS E2EE mã hóa message, một relay node độc hại không đọc được nội dung
	•	Sender identity: An toàn — Sealed Sender, node relay không biết who-to-whom
	•	Metadata: Có rủi ro — attacker có thể thấy device ID, message size, timing pattern trên BLE
	•	Identity commitment: CẦN VÁ NGAY — BitChat dùng HMAC(R, PK)[0:8] chỉ 64 bit, có thể brute-force trong môi trường offline

Điểm yếu cần vá so với BitChat
CRITICAL: Thay truncated 8-byte identity commitment bằng full 32-byte Ed25519 public key fingerprint trong BLE beacon. Kết hợp với challenge-response protocol trước khi accept peer vào mesh. Chi phí: thêm 2-3 giây latency khi peer discovery — hoàn toàn chấp nhận được.

Kiến trúc Hybrid KEM được khuyến nghị
Implement ML-KEM-768 (NIST FIPS 203) song song với X25519 theo kiến trúc Hybrid KEM. Cách tiếp cận này là "buy once, enable when needed" — implement một lần, control per-tenant theo license tier:

// Hybrid KEM: cả hai phải bị break mới compromise
Final_Session_Key = HKDF(
    X25519_shared_secret ||
    ML_KEM_768_shared_secret
)

// Overhead per handshake:
//   Bandwidth: +1.2 KB (Kyber ciphertext)
//   Latency:   +3-5 ms trên Apple Silicon
//   Acceptable cho enterprise messaging

Chiến lược triển khai theo tier:

	•	Enterprise Standard tier: X25519 only — PQ disabled mặc định. Có thể bật theo yêu cầu.
	•	Gov/Military tier: Hybrid KEM bật mặc định, không thể tắt. Ghi vào License JWT.
	•	EMDP Key Escrow trong Mesh: Upgrade từ ECIES/Curve25519 thuần sang Hybrid ECIES + ML-KEM-768. Đây là gap lớn nhất hiện tại trong thiết kế EMDP.


PHẦN 3: SLA & LOCAL AI AUTO-FIX
3.1  Làm rõ mô hình — Phân tích lại theo scope đúng
Sau khi làm rõ scope, mô hình bạn mô tả không phải là anti-pattern — nó là một thiết kế lành mạnh, rất gần với cách Claude Code hoạt động: AI phân tích, đề xuất, và human giữ quyền quyết định cuối cùng cho mọi action có rủi ro.

Nguyên tắc cốt lõi: AI không có execution authority — chỉ có information authority. Mọi action vượt ngoài read-only operations đều cần human approve. Đây là boundary không được phép mờ.

3.3  Fine-tuning từ Community Contributions
Đây là điểm thú vị nhất của thiết kế. Một số yêu cầu kỹ thuật quan trọng cần xem xét:

Sanitization Pipeline — bắt buộc trước khi đưa vào training
	•	PII Stripping: Bug reports thực tế chứa stack trace với session ID, memory address, đôi khi partial message content. Cần strip toàn bộ trước khi đưa vào training set — yêu cầu pháp lý với enterprise customer.
	•	Structured Error Format: Thiết kế một structured error reporting format trong TeraChat client: error_type, sanitized_stack_trace, OS_version, TeraChat_version, CRDT_state (không có message content). Data này clean hơn nhiều so với free-form bug reports.
	•	Community contribution flow: Developer / community submit error report → review và sanitize → add vào training corpus → trigger fine-tuning theo schedule (không trigger real-time)

3.4  Guardrails bắt buộc — Ranh giới không được mờ
TUYỆT ĐỐI KHÔNG cho AI access vào: HSM / signing key, OPA policy engine, MLS epoch rotation trigger, production database write path, binary execution environment. Những hành động này phải có human signature.

Danh sách action theo risk level:

	•	[AUTO-EXECUTE — LOW RISK]: Restart crashed service, clear corrupted sled cache, WAL integrity check, vacuum trigger, rotate log files
	•	[HUMAN APPROVE — MEDIUM RISK]: Rollback database đến WAL checkpoint cụ thể, failover sang passive node, thay đổi config parameter
	•	[HUMAN APPROVE + SECOND APPROVAL — HIGH RISK]: Schema migration, dependency update, OPA policy suggestion (chỉ đề xuất, không apply)
	•	[AI KHÔNG ĐƯỢC ĐỀ XUẤT]: Bất cứ thứ gì liên quan đến key material, cryptographic operation, hay security policy enforcement
