Mục Đích & Giá Trị Cốt Lõi

TeraChat được định vị là Sovereign Work OS — nền tảng cộng tác doanh nghiệp bảo mật đầu cuối (E2EE), hướng đến các môi trường yêu cầu khắt khe: tài chính, chính phủ, quốc phòng, y tế tại Việt Nam và Đông Nam Á. Khác biệt cốt lõi nằm ở 3 cam kết kiến trúc bất biến:

Cam kết 1
🔐 Zero-Knowledge Architecture
Server relay là "blind router" — chỉ thấy ciphertext blob. Không có key giải mã, không biết danh tính người gửi (Sealed Sender), không đọc nội dung. Đảm bảo bằng MLS RFC 9420 + hardware-bound key management.
Cam kết 2
📡 Offline Survival Capability
BLE 5.0 + Wi-Fi Direct Mesh tự động kích hoạt khi mất Internet. CRDT DAG đảm bảo đồng bộ cuối cùng. Không nền tảng enterprise nào có khả năng này — đây là moat kỹ thuật thực sự.
Cam kết 3
⚡ Operational Simplicity
1 Rust binary, 5-minute deploy, $6–48/month VPS. Không Kubernetes, không PostgreSQL cluster, không DevOps team. Cấu trúc chi phí infra $7–48/tenant/month (vs $200–500 đối thủ).

Nhóm Crypto & E2EE
MLS RFC 9420 — Message Layer Security

TreeKEM O(log n) key distribution cho group ≤ 10,000 members. Epoch rotation khi member rời group (immediate) hoặc theo lịch 24h. Forward Secrecy tự động.

Hybrid PQ-KEM (CNSA 2.0 compliant)
Final_Session_Key = HKDF(X25519_Shared || Kyber768_Shared)
-- X25519: classical ECDH (< 100 bytes, fast handshake for SOS)
-- ML-KEM-768: post-quantum KEM, payload ~1.18KB
-- PQ payload chỉ kèm vào KeyPackage hoặc mỗi 10,000 messages
-- BLE Mesh: RaptorQ FEC fragmentation thành ≤ 400 byte chunks

OOB Symmetric Push Ratchet: Push_Key là HKDF one-way chain độc lập với MLS TreeKEM. NSE giải mã O(1) RAM (< 5MB). NSE không bao giờ tái dựng TreeKEM — đây là isolation đúng đắn.

ZeroizeOnDrop — Memory Zeroization Protocol 
-- Mandatory cho mọi struct giữ key material
#[derive(ZeroizeOnDrop)]
struct EpochKey([u8; 32]);

-- iOS: Double-Buffer Zeroize (Jetsam mitigation)
-- Phân bổ key vào 2 page MAP_ANONYMOUS|MAP_PRIVATE
-- Ghi đè 0x00 vào page 1 TRƯỚC KHI dùng key (page 2)
-- ZeroizeOnDrop: ghi đè cả 2 page
-- Android: Wrap-on-Derive (Doze mitigation)
-- plaintext_key tồn tại < 1ms (derive → StrongBox wrap đồng bộ)

Nhóm Network & Protocol
ALPN Negotiation — Adaptive Protocol Selection 
Step 1: QUIC/HTTP3 over UDP:443         → ACK < 50ms → ONLINE_QUIC (~30ms RTT)
         │ Timeout/DROP
Step 2: gRPC/HTTP2 over TCP:443         → TLS OK → ONLINE_GRPC (~80ms RTT)
         │ DPI block
Step 3: WebSocket Secure over TCP:443   → WS OK → ONLINE_WSS (~120ms RTT)
         │ All fail
→ MESH_MODE

Total negotiation: < 50ms (parallel probe, not sequential)
Adaptive learning: probe_fail_count ≥ 3 → strict_compliance = true
                   Notify Admin: "Auto-switched to TCP on this network"

Bug đã được xác định (NI-01 trong Arrange.md): Logic cũ increment probe_fail_count TRƯỚC khi race condition resolve — dẫn đến strict_compliance bị set sai. Fix: chỉ increment khi QUIC xác nhận failed và không phải winner.

5.3 — Error Handling Flows
Remote Wipe — Non-interruptible sequence 
Trigger: self.userID in removedMembers of MLS Commit

Xác Thực & Phân Tích Rủi Ro

15 vấn đề được xác định, phân loại theo severity và impact thực tế.

Phương pháp: Các vấn đề sau được trích xuất từ Arrange.md conflict audit (15 conflicts) và cross-reference validation giữa 9 spec files. Phân loại: Blocker (ship-stopper) / High (pre-production) / Medium / Low.

6.1 — BLOCKER Issues (3) — Không ship được nếu chưa fix
WasmParity CI Gate — Chưa implement
wasm3 (iOS) vs wasmtime (Android/Desktop) có thể cho output semantics khác nhau. Không có CI gate = không phát hiện divergence cho đến khi user báo lỗi sản xuất. Core_Spec §11.4 ref FI-03.
BLOCKER
Implement: cargo test --test wasm_parity -- --timeout 60. Criteria: semantic identity 100%, latency delta ≤ 20ms, memory delta ≤ 5MB. Block merge on fail.
CI/CD Code Signing Pipeline — 5 platforms chưa có
Windows EV Code Signing, Linux GPG/Cosign, iOS fastlane match, Android Keystore chưa được automated. Không có signing = SmartScreen blocks install trên Windows, deb/rpm không verify được trên Linux.
BLOCKER
ops/signing-pipeline.md cần implement đầy đủ. Windows EV cert: mua tháng 1 (30-day reputation build cần thiết). Linux: dpkg-sig + cosign. iOS: fastlane match.
Dart FFI NativeFinalizer Clippy Lint — Chưa có
Không có lint rule = developer có thể để GC làm finalizer cho TeraSecureBuffer. GC release không phải primary path — là security smell (key material có thể tồn tại lâu hơn cần). PLATFORM-17 violation.
BLOCKER
Implement tera_avoid_direct_ffi_pointer và tera_require_secure_buffer Dart lints. Enforce via tera_dart_lints analyzer plugin. GC release → WARNING metric only (không thể prevent hoàn toàn nhưng phải track).
6.2 — HIGH Risk Issues (7)
ALPN Parallel Probe Race — probe_fail_count corruption (NI-01)
Logic cũ increment probe_fail_count TRƯỚC khi tokio::select! resolve. Nếu cả QUIC và gRPC probe đều succeed, count tăng sai → strict_compliance bị bật trên network tốt.
HIGH
Fix: chỉ increment khi QUIC confirmed failed và không phải winner của race. Code đã có trong Arrange.md EDIT C-10.
Android Doze Mid-ZeroizeOnDrop (RC-02)
Android không expose mlock() API. Process freeze mid-ZeroizeOnDrop có thể để key material trong RAM page chưa zero-fill. Xảy ra khi Doze kill process giữa key derivation và wrap vào StrongBox.
HIGH
Wrap-on-Derive pattern: derive → StrongBox wrap đồng bộ trong < 1ms, không có await point giữa 2 bước. Doze không thể freeze ở giữa window này.
NSE RAM Budget Additive Overflow (RC-03)
NSE Arena 10MB + MLS decrypt ~2MB + system overhead ~2MB = 14MB baseline. ONNX Micro-NER load thêm 8MB = 22MB > 20MB ceiling → Jetsam kill → notification mất silently. Không có error log.
HIGH
NsePolicy::ProhibitOnnxLoad bắt buộc. NSE chỉ dùng regex-only PII detection (Aho-Corasick). Full NER defer sang Main App qua main_app_decrypt_needed flag.
EMDP Shun + MLS Epoch Causal Deadlock (BL-02)
Trong all-iOS EMDP mesh, nhận Shun Record cho node thông thường → cần evict nhưng không thể rotate MLS Epoch (EMDP không cho phép). Deadlock: node bị compromise nhưng vẫn trong group key.
HIGH
EMDP Shun Exception Protocol: Case A (regular member) → TempGroupKey từ HKDF(EmdpKeyEscrow.relay_session_key, "shun-rekey"||poisoned_node_id). Case B (Tactical Relay bị shun) → terminate EMDP session immediately.
OPA Policy + Federation Schema Lock (BL-03)
Khi federation ở READ_ONLY mode (±1 minor schema delta), OPA policy updates bị block. Branch cluster có thể stuck với stale (insecure) OPA policy vô thời hạn.
HIGH
OPA Policy distribution channel độc lập với data federation channel. Policy sync EXEMPT khỏi schema restrictions — security update phải đến được mọi cluster bất kể schema version gap.
AppArmor/SELinux Startup Crash (Linux)
Không có postinstall MAC profile = startup crash trên memfd_create và ipc_lock denial trên enforcing Ubuntu/RHEL systems. Gov/Military thường dùng enforcing SELinux — đây là showstopper.
HIGH
Feature_Spec.md §16 postinstall script. Self-test: terachat --check-permissions. AppArmor: apparmor_parser; SELinux: semodule -i. Detection at install time, not runtime.
Dead Man Switch Audit Trail Gap (BL-01)
CallKit active call defer Dead Man Switch — hành vi đúng nhưng không có audit log. CISO không thể phân biệt legitimate deferral vs compromise. Cho Gov/Military là compliance gap.
HIGH
DeadManDeferralEntry struct với device_counter, server_counter delta, call_id. Counter delta giúp CISO assess risk. Ed25519-signed, append-only.
6.3 — MEDIUM Risk Issues (5)
BLE Channel Confusion — PQ Key Exchange
Spec không phân biệt rõ BLE Advertising channel (31 bytes) vs GATT Connected channel (512 bytes). ML-KEM-768 payload ~1.18KB không fit vào ADV_EXT_IND PDU.
MEDIUM
Clarify: ADV_EXT_IND = peer discovery only (HMAC commitment). ML-KEM exchange dùng GATT connected mode với RaptorQ FEC. Không fragment key material vào beacon frames.
VACUUM INTO Concurrent Write Race (NI-03)
VACUUM INTO hot_dag_tmp.db không được wrap trong EXCLUSIVE TRANSACTION. Concurrent Tokio writer có thể swap incomplete DB qua atomic rename().
MEDIUM
Prefer WAL checkpoint thay VACUUM cho append-only DB. Nếu VACUUM INTO cần thiết: BEGIN EXCLUSIVE TRANSACTION trước. Code fix trong Arrange.md EDIT C-09.
Huawei WASM JIT vs AOT Ambiguity (RC-01)
Spec không rõ runtime mode: AppGallery yêu cầu AOT .waot bundle nhưng runtime có thể vẫn dùng JIT. Gây nhầm lẫn cho developer và QA.
MEDIUM
Clarify: AOT bundle = AppGallery review requirement. Runtime = JitWithAotFallback mode. Primary: wasmtime JIT; Fallback: AOT khi JIT detection fails.
Border Node Auto-Detection — Algorithm Undefined
Border Node được mô tả là "auto-assigned khi internet_available && ble_active" nhưng heuristics cụ thể chưa được spec. Không rõ debounce, hysteresis, priority.
MEDIUM
Cần define: debounce window (5s), check interval, priority khi nhiều devices đều eligible, handoff protocol khi border node rời mesh.
Windows ARM64 SAB Validation — Chưa test
WebView2 trên ARM64 Windows (Surface Pro X, Copilot+ PCs) có thể có SAB behavior khác x64. Chưa có test coverage → có thể fallback sang Named Pipe silently mà không log.
MEDIUM
Pre-release SAB validation trên ARM64 WebView2. Đảm bảo tier selection được log đúng (IPC-02 — silent tier change = BLOCKER).
6.4 — Điểm Mạnh Kiến Trúc Đã Xác Thực
  • Zero orphan features: 15/15 features có verified Core module mapping — không có floating feature nào.
  • Token Protocol FFI: Thiết kế opaque u64 token thay vì raw pointer là đúng và an toàn. No raw pointer in pub extern C — enforced by Clippy.
  • Two-DB safety net: cold_state.db luôn rebuildable từ hot_dag.db — migration failures không gây data loss.
  • Stateless VPS relay invariant: EMDP journal trong hot_dag.db (client) không phải trên relay — preserves zero-knowledge.
  • Crash-safe WAL: SQLite WAL replay on next open — power loss không mất data.

7.2 — Cải Tiến Kỹ Thuật Cụ Thể
🔧 Simplification: ONNX Inference Backend Selection

Hiện tại resolve_inference_backend() có 5 tầng với logic phức tạp. Đề xuất làm rõ priority order và fail-fast behavior:

Priority (enforced, not preference):
1. TeraEdge Desktop LAN (mDNS, latency < 50ms, load < 80%) → RemoteDesktop
2. Local model (RAM sufficient AND battery > 30%) → Local
3. VPS Enclave (internet available) → VpsEnclave
4. Downgrade model tier + retry → LowerTierLocal
5. All unavailable → Unavailable (không phải error, feature gracefully absent)

Key clarification cần thêm vào spec:
- "RAM sufficient" = available_ram_mb > 100 (Tiny) or > 200 (Base)
- Thermal state critical → skip LocalBattery, jump to step 3
- Desktop offline mid-inference: retry once (3s timeout) → then VPS or Tiny local
🔧 Cải tiến: IPC Bridge Dart FFI Transaction Timeout

Hiện tại khi TTL timeout, emit IpcSignal::TransactionTimeout và UI hiển thị "Session expired". Nhưng không rõ: (1) TTL bao lâu? (2) Ai set TTL? (3) Retry policy?

Đề xuất làm rõ:
- TTL mặc định: 30s cho media operations, 5s cho text operations
- Set bởi Rust Core (không phải UI — UI không biết TTL)
- Timeout flow:
    tera_buf_acquire(op_id) → token valid
    [30s elapsed, no releaseNow()]
    → Rust Core emits IpcSignal::TransactionTimeout
    → token revoked
    → UI displays "Session expired — please retry"
    → GC Finalizer sẽ catch nếu buffer vẫn referenced (WARNING metric)

Không cần retry tự động — retry là trách nhiệm của UI action (user retriggers)
🔧 Cải tiến: EMDP Key Escrow — Tainted Escrow Long-term Fix

Khi Shun xảy ra trong EMDP và poisoned node đã nhận EmdpKeyEscrow, toàn bộ EMDP window messages bị tainted. Workaround hiện tại (TempGroupKey) là accepted risk. Long-term fix:

Đề xuất (Sprint 3 / TD-01):
Desktop pre-provision N escrow keys trước khi offline:
  EmdpKeyEscrow_1 = ECIES.encrypt(iOS_device_1_pubkey, relay_session_key_1)
  EmdpKeyEscrow_2 = ECIES.encrypt(iOS_device_2_pubkey, relay_session_key_2)
  ...

Nếu Tactical Relay bị Shun:
  Next highest-battery iOS device nhận EmdpKeyEscrow_N → assume role
  Không cần TempGroupKey workaround
  Không có tainted escrow risk

Implementation note: Desktop cần biết trước identity của potential relays
→ BLE peer discovery phải chạy TRƯỚC khi Desktop offline
🔧 Đơn giản hóa: License Validation Strategy

Cryptographic entanglement (DeviceIdentityKey + LicenseToken + CurrentEpoch → MasterUnlockKey) là thiết kế thông minh nhưng tạo ra coupling chặt. Đề xuất kiểm tra thực tế:

Scenario analysis:
- License JWT expired → MasterUnlockKey sai → cold_state.db = rác → User bị lock
  → Q: Có recovery path không nếu license renewal bị trễ?
  → A hiện tại: Grace period T+0 chat/mesh continue. Đủ.

- Device re-enrollment → DeviceIdentityKey mới → MasterUnlockKey mới
  → cold_state.db cũ không decrypt được
  → Q: recovery từ hot_dag.db rebuild có đủ không?
  → A: Có, đây là DB-02 safety net. OK.

- Epoch rotation → MasterUnlockKey thay đổi nếu CurrentEpoch là input
  → Q: MasterUnlockKey update mỗi epoch hay one-time?
  → CẦN LÀM RÕ: Nếu per-epoch, mỗi epoch rotation cần re-derive master key
    và có thể cần re-encrypt cold_state.db. Chi phí này cần được documented.
🔧 Bổ sung thiếu: CRDT Tombstone GC Policy Explicit Definition

Tombstone GC chỉ xảy ra khi tombstone.clock ≤ MVC (Merged Vector Clock). Nhưng: (1) MVC được cập nhật như thế nào và khi nào? (2) Mesh partition kéo dài bao lâu thì MVC bị stale? (3) 7-day hot DAG eviction suspension khi Mesh active — rõ ràng. Nhưng 7 ngày được tính từ khi nào (message created hay last merge)?

Cần làm rõ trong spec:
- MVC update trigger: sau mỗi Gossip round khi ≥ M peers ACK
- MVC staleness threshold: không GC nếu bất kỳ peer nào offline > 48h
- 7-day eviction: tính từ tombstone.clock (event time), không phải server receipt time
- Emergency override: CISO có thể force GC via Admin Console (với audit log)

7.3 — Guardrails Không Được Thay Đổi

Các quyết định kiến trúc bất biến — không được thay đổi dù có lý do gì:

  • Native Crypto Offloading: WASM không tự chạy crypto. Mọi op qua Rust Core Host ABI. 150–500× throughput advantage. Không exception.
  • wasm3 trên iOS: iOS W^X policy prohibits JIT. wasm3 pure interpreter là giải pháp duy nhất. WasmParity CI gate validate behavioral equivalence với wasmtime.
  • Stateless VPS relay: EMDP journal trong client hot_dag.db. Không lưu journal server-side — vi phạm zero-knowledge và stateless invariant.
  • iOS election_weight = 0: iOS không bao giờ là Mesh Dictator. Jetsam có thể kill process bất kỳ lúc nào — không reliable làm DAG merge authority.
  • Enterprise-only positioning: Không consumer pricing, không public marketplace, không self-service signup. B2B annual contract model là duy nhất.
  • ring crate only for crypto: Không self-implement bất kỳ cryptographic primitive nào. Không swap sang alternative crypto crates mà không extensive security review.
